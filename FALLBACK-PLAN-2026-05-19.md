# План: безопасный fallback на роутере после XHTTP-фейла

> Контекст: 2026-05-19 после неудачной XHTTP-миграции единственный аплинк = gRPC :2053.
> Если он сдохнет — VPN ляжет. На ноуте есть AmneziaWG, на других LAN-клиентах ничего.
> Нужен хоть какой-то fallback, минимум риска для текущего рабочего состояния.

## Принципы

1. **Не трогать рабочий gRPC.** Каждое изменение прода — потенциальный 1-минутный
   silent-failure (как XHTTP только что). gRPC останется тегом `proxy-grpc` и
   единственным членом `selector` балансера, пока не подтвердим что fallback РЕАЛЬНО
   работает через tproxy.
2. **Любой новый outbound — тегом БЕЗ префикса `proxy-`.** Observatory его не подхватит,
   balancer не включит. Проверяем явным routing-правилом на тестовый домен.
3. **Любой новый outbound тестируем end-to-end** (LAN-клиент → tproxy → outbound → инет),
   а не только observatory probe или изолированный SOCKS5.
4. **Все изменения через `xray-apply-config.sh`** с авто-rollback. Промоут в safe.json
   **только** после end-to-end подтверждения.
5. **Сначала ручной флип, потом автоматика.** Цель — иметь fallback который можно
   включить одной командой за 10 секунд, не автоматическое переключение которое может
   уронить юзера в silent failure.

## Уровни fallback (от безопасного к рискованному)

### Уровень 0 — что уже есть (бесплатно)

- AmneziaWG на ноуте → если роутер встал, ноут руками переключается на него.
- Прямые VLESS-клиенты на ноуте → то же самое.
- В iOS у пользователя возможно есть импортированный VLESS-клиент.

**Действие:** ничего не делать, проверить что AmneziaWG-конфиг на ноуте свежий и
подключается. Этого может хватить если роутерный fallback откладывать.

### Уровень 1 — резервный gRPC inbound на сервере (самый безопасный)

**Идея:** на сервере уже есть рабочий gRPC inbound id=5 :2053. Создать второй gRPC
inbound на **другом порту** (например :2054) с **другим serviceName**. На клиенте
добавить тегом `grpc-backup` (не `proxy-*`). Если основной gRPC сдохнет — флипнуть
руками: переименовать `grpc-backup` → `proxy-grpc-backup` (чтобы observatory его взял)
ИЛИ заменить serverPort/serviceName в основном `proxy-grpc`.

**Плюсы:**
- Транспорт **тот же что и рабочий** — никакого нового кода в xray.
- Если DPI на провайдере таргетирует :2053 конкретно — :2054 может пройти.
- tproxy+gRPC мы уже проверили в проде, не silent-failure-prone как XHTTP.
- Серверная часть тривиальна (создать ещё один inbound через 3x-ui).

**Минусы:**
- Если DPI таргетирует **транспорт gRPC** (а не порт), оба сдохнут одновременно.
- Не помогает если порт :2053 заблокируют на провайдере.

**Риск:** низкий. Конфликтов с tproxy нет, мы уже знаем что gRPC работает.

### Уровень 2 — VLESS+TCP+Reality на ДРУГОМ порту (средний)

**Идея:** на сервере есть Reality-inbound id=??? :45865 (тот, что работает с телефона).
Использовать его параметры с роутера — но не в balancer'е, а в холодном резерве.

**Плюсы:**
- Серверная часть готова, ничего не делать.
- Reality маскирует трафик под чужой TLS-домен, DPI его не трогает по портам
  обычно.

**Минусы:**
- С роутера тот же inbound уже пытались — 4/5 probes ОК, 1/5 timeout. Не идеально
  стабилен, но **РАБОТАЕТ** (трафик доходил). Это сильно лучше XHTTP который не работал
  совсем.
- Reality+TCP+Vision с tproxy мы проверили в прошлой сессии — silent-failure нет,
  только частичная нестабильность.

**Риск:** низкий. Худший случай — 20% запросов медленные.

### Уровень 3 — XHTTP на отдельном tproxy (высокий риск)

**Идея:** разобраться почему XHTTP+tproxy зависает (отдельная исследовательская задача).
См. план «Что НЕ делать сразу» в EXTRA_CARE.md инциденте 2026-05-19.

**Риск:** высокий, и не нужен пока есть Уровень 1/2.

### Уровень 4 — AmneziaWG на роутере (радикальный)

**Идея:** установить amneziawg-go или wireguard-go на роутер, отдельный туннель,
маршрут для трафика когда xray мёртв.

**Риск:** очень высокий — новый сервис, новые правила маршрутизации, может конфликтовать
с tproxy/nft. **Не делать пока нет реальной необходимости.**

## Рекомендуемый порядок действий

### Этап А (сейчас) — без изменений на роутере

1. **Зафиксировать что AmneziaWG на ноуте действительно работает.** Проверь сейчас:
   - Открой AmneziaGuard, подключись.
   - Проверь что инет работает (https://www.gstatic.com/generate_204 = 204).
   - Это и есть твой текущий аварийный fallback пока на роутере что-то делаем.

2. **Обновить репо**: коммит с `EXTRA_CARE.md` инцидентом + `cur-config.json` снапшот
   + этот план. БЕЗ изменений в production-конфигах.

### Этап Б (когда будет час спокойствия) — Уровень 1, gRPC backup

3. **Сервер**: создать ещё один gRPC inbound через 3x-ui:
   - port: `:2054` (или `:8443` — порт от старого WS, он свободен и стандартен для https)
   - serviceName: например `grpc-mordepy-backup`
   - тот же UUID, тот же сертификат, ALPN h2

4. **Изолированный тест с роутера**: написать аналог `xray-test-xhttp-isolated.sh`
   но для gRPC — `xray-test-grpc-isolated.sh`. Запустить, убедиться что httpbin.org
   видит IP сервера. Это **single-purpose тест клиентского gRPC-выхода**.

5. **На роутере**: добавить outbound тегом `grpc-backup` (не `proxy-grpc-backup`!)
   с параметрами нового сервера. Балансер не трогать. Добавить временное routing-правило
   `outboundTag: grpc-backup, domain: ["domain:httpbin.org"]`. Применить через
   `xray-apply-config.sh` с rollback 10 мин.

6. **End-to-end тест с LAN-клиента** (ноут):
   ```
   curl -sS https://httpbin.org/ip  # должен показать IP сервера
   ```
   Если работает с ноута через tproxy → XHTTP-инцидент не повторится, gRPC+tproxy
   мы знаем.

7. **Промоут в safe**: убрать временное routing-правило, оставить outbound `grpc-backup`
   как cold-standby (без префикса proxy-, не в селекторе). Промоут в safe.json.

8. **Документировать как флипнуть:** одна команда что меняет `proxy-grpc` → новые
   параметры в случае падения основного. Положить как `/root/xray-flip-to-backup.sh`
   (переживёт ребут).

### Этап В (опционально потом) — Уровень 2, Reality cold-standby

Когда Уровень 1 будет работать как cold-standby — повторить аналогично с Reality :45865.
Получим **два** независимых fallback'а на разные порты и транспорты.

### Этап Г (намного позже) — Уровень 3, XHTTP

Только когда Уровни 1+2 в холодном резерве и хочется идеала. Сначала исследовать
почему XHTTP+tproxy ломается (loglevel=debug, tcpdump).

## Что НЕ делать прямо сейчас

- ❌ Не трогать `proxy-grpc` в `selector` балансера.
- ❌ Не добавлять новый outbound в `selector` балансера до end-to-end теста с LAN.
- ❌ Не использовать `enableConcurrency=false` с >1 outbound в selector.
- ❌ Не доверять observatory probe как единственному признаку «alive».
- ❌ Не делать одновременно сервер + клиент + balancer изменения. Шаг за шагом.

## Verification end-to-end (для каждого этапа)

Один и тот же чек-лист до промоута в safe.json:
1. С ноута `curl https://httpbin.org/ip` → IP сервера, latency < 1s.
2. С роутера через 127.0.0.1:8118 — 8/8 probes < 0.5s.
3. error.log без новых ошибок про новый outbound в течение 5 мин.
4. `xray api bi vpn-balancer` — selects стабильный (не флапает).
5. TG-сообщения от selfheal/balancer-watch — без неожиданных событий.
6. Браузерный smoke: youtube/claude/github открываются.
